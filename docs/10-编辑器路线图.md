# 10 — 编辑器路线图

本文档描述 `@agile-sofl/parser` 与 **LSP 语言服务**、**Hybrid 编辑器** 的对接思路。第一期 LSP 与 VS Code 扩展已实现，详见 [11-VSCode扩展与LSP.md](./11-VSCode扩展与LSP.md)。

## 1. 目标场景

| 能力 | 用户价值 |
|------|----------|
| 实时诊断 | 编写时发现语法/语义错误 |
| 跳转定义 | 模块、类型、变量、进程间导航 |
| 悬停信息 | 符号类型、FSF 摘要 |
| 格式化 | 统一规格排版 |
| FSF 风格提示 | `ASFL_FSF_*` 规约建议 |
| 结构大纲 | 模块树、进程列表 |

## 2. 已有可复用 API

```typescript
check(source)                          // 完整检查
parse(source)                          // 仅解析（增量场景更快）
inspect(source, options?)              // 人类可读报告
format(source)                         // 格式化
walk(ast, visitor)                     // AST 遍历
getNodeAtOffset(ast, offset)           // 光标定位节点
resolveScope(program)                  // 符号表
formatDiagnostic(d, source?)           // 诊断格式化
DiagnosticCodes                        // ASFL_* 稳定码
```

### 2.1 Span 与 LSP 位置

```typescript
interface Span { start: number; end: number; line: number; column: number }
```

LSP `Position` ↔ `Span`：建议 UTF-16 码元（与 VS Code 一致）。`Diagnostic.span` → LSP `Diagnostic.range`。

### 2.2 诊断码 → LSP severity

| 库码 | LSP |
|------|-----|
| `ASFL_PARSE_001`, `ASFL_LEX_001`, `ASFL_SCOPE_*`, `ASFL_TYPE_*` | Error |
| `ASFL_FSF_001` | Warning |
| `ASFL_FSF_002`, `ASFL_FSF_003` | Information |

## 3. LSP 架构草案

```text
Editor ──LSP──► agile-sofl-language-server
                  ├─ debounce → check()
                  ├─ publishDiagnostics
                  ├─ formatting → format()
                  ├─ definition / hover / documentSymbol
                  └─ @agile-sofl/parser
```

**第一期**：diagnostics + formatting + documentSymbol（`walk` 收集 module/process/function）。

**第二期**：definition（`getNodeAtOffset` + `lookupSymbol`）、hover、workspace/symbol。

**第三期**：增量解析、completion、FSF snippet。

## 4. walk 访问器扩展

当前 `src/visitor/walk.ts` 覆盖 program/module/process/function/expression。

建议扩展：

```typescript
interface Visitor {
  enterTypeDecl?(node: TypeDeclNode): void
  enterVarDecl?(node: VarDeclNode): void
  enterFsfSpec?(node: FsfSpecNode): void
  enterInformalText?(node: InformalTextNode): void
}
```

用途：documentSymbol 大纲、informal 区域语义高亮。

## 5. getNodeAtOffset 增强

当前实现遍历 module/process/function/expression，后访问者覆盖，**未选最内层节点**。

改进计划：

1. 深度优先遍历所有带 span 节点
2. 选包含 offset 且 span 最小的节点
3. 返回类型化 `ASTNode | null`

```typescript
function findSmallestContainingNode(ast, offset): ASTNode | null {
  let best = null, bestLen = Infinity
  visitAllNodes(ast, node => {
    const { start, end } = node.span
    if (offset >= start && offset <= end && end - start < bestLen) {
      best = node; bestLen = end - start
    }
  })
  return best
}
```

此为 go-to-definition 与 hover 的核心。

## 6. Hybrid 编辑器集成

Hybrid = 形式 FSF + 非形式 comment/informal 文本。UI 分层：

```text
process A
┌─ FSF (formal) ──────────────┐
│ x > y && ... => ...         │
│ others && ...               │
└─────────────────────────────┘
┌─ comment (informal) ────────┐
│ informal note               │
└─────────────────────────────┘
```

路径：解析 AST → `walk` 定位 `fsf_spec` / `informal_text` / `comment` → 按 span 装饰编辑器 → `ASFL_FSF_*` 波浪线。

格式化 MVP 用全量 `format()`；长期考虑增量 print 保留 informal 自由文本。

## 7. 性能

| 策略 | 说明 |
|------|------|
| Debounce | 200–400ms 后触发 check |
| 单模块模式 | `parseSingleModule` 减开销 |
| 缓存 | sourceHash → CheckResult |
| 分级检查 | 解析 error 时跳过 type/fsf（库已支持） |

banking fixture 可作为 <100ms 性能基准（典型规格 <2000 行）。
