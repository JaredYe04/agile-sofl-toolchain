# 10 — 编辑器路线图

本文档描述 `@agile-sofl/parser` 与 **LSP 语言服务**、**Hybrid 编辑器** 的对接思路。第一期 LSP 与 VS Code 扩展已实现，详见 [11-VSCode扩展与LSP.md](./11-VSCode扩展与LSP.md)。

## 1. 目标场景

| 能力 | 用户价值 |
|------|----------|
| 实时诊断 | 编写时发现语法/语义错误 |
| 跳转定义 | 模块、类型、变量、进程间导航 |
| 悬停信息 | 符号类型、FSF 摘要 |
| 格式化 | 统一规格排版 |
| FSF 风格提示 | `ASFL_FSF_*` 规约建议 |
| 结构大纲 | 模块树、进程列表 |

## 2. 已有可复用 API

```typescript
check(source)                          // 完整检查
parse(source)                          // 仅解析（增量场景更快）
inspect(source, options?)              // 人类可读报告
format(source)                         // 格式化
walk(ast, visitor)                     // AST 遍历
getNodeAtOffset(ast, offset)           // 光标定位节点（findNodeAtOffset 最内层）
findNodeAtOffset(ast, offset)          // 推荐：最小 span 节点
resolveReference(program, scope, off)  // 引用 → 声明
resolveScope(program)                  // 符号表
formatDiagnostic(d, source?)           // 诊断格式化
DiagnosticCodes                        // ASFL_* 稳定码
```

### 2.1 Span 与 LSP 位置

```typescript
interface Span { start: number; end: number; line: number; column: number }
```

LSP `Position` ↔ `Span`：建议 UTF-16 码元（与 VS Code 一致）。`Diagnostic.span` → LSP `Diagnostic.range`。

### 2.2 诊断码 → LSP severity

| 库码 | LSP |
|------|-----|
| `ASFL_PARSE_001`, `ASFL_LEX_001`, `ASFL_SCOPE_*`, `ASFL_TYPE_*` | Error |
| `ASFL_FSF_001` | Warning |
| `ASFL_FSF_002`, `ASFL_FSF_003` | Information |

## 3. LSP 架构草案

```text
Editor ──LSP──► agile-sofl-language-server
                  ├─ debounce → check() [cached by version + hash]
                  ├─ publishDiagnostics (incl. ASFL_SCOPE_001)
                  ├─ formatting → format()
                  ├─ definition / hover / completion / folding
                  └─ @agile-sofl/parser
```

**第一期（已完成）**：diagnostics + formatting + documentSymbol + semanticTokens。

**第二期（已完成）**：definition（`findNodeAtOffset` + `resolveReference`）、hover、completion（类型/符号/FSF snippet）。

**第三期（已完成）**：诊断结果缓存（document version + content hash）、`ASFL_SCOPE_001` 实时诊断、函数/comprehension 绑定 hover 与补全、workspace/symbol、folding。

**第四期（规划中）**：增量解析、FSF 高级 snippet。

## 4. walk 访问器扩展

当前 `src/visitor/walk.ts` 覆盖 program/module/process/function/expression。

建议扩展：

```typescript
interface Visitor {
  enterTypeDecl?(node: TypeDeclNode): void
  enterVarDecl?(node: VarDeclNode): void
  enterFsfSpec?(node: FsfSpecNode): void
  enterInformalText?(node: InformalTextNode): void
}
```

用途：documentSymbol 大纲、informal 区域语义高亮。

## 5. 光标定位（已完成）

`findNodeAtOffset(ast, offset)` 经 `visitAllNodes` 选取 **span 最小且包含 offset** 的节点，供 definition / hover 使用。`getNodeAtOffset` 保留为别名。

`collectHybridRegions(ast)` 收集 FSF / informal / comment / decom 区域，供 Hybrid 装饰 POC 使用。

## 6. Hybrid 编辑器集成

Hybrid = 形式 FSF + 非形式 comment/informal 文本。UI 分层：

```text
process A
┌─ FSF (formal) ──────────────┐
│ x > y && ... => ...         │
│ others && ...               │
└─────────────────────────────┘
┌─ comment (informal) ────────┐
│ informal note               │
└─────────────────────────────┘
```

路径：解析 AST → `walk` 定位 `fsf_spec` / `informal_text` / `comment` → 按 span 装饰编辑器 → `ASFL_FSF_*` 波浪线。

格式化 MVP 用全量 `format()`；长期考虑增量 print 保留 informal 自由文本。

## 7. 性能

| 策略 | 说明 |
|------|------|
| Debounce | 200–400ms 后触发 check |
| 单模块模式 | `parseSingleModule` 减开销 |
| 缓存 | `diagnosticsForDocument`：document version + SHA-256 content hash → CheckResult |
| 分级检查 | 解析 error 时跳过 type/fsf（库已支持） |
| 基准脚本 | `npm run bench:parse`；CI 对 banking 断言 `--assert-p95=100`（阻塞） |

`tests/fixtures/integration/banking.asfl` 与 `examples/ecommerce.asfl` 为典型集成基准；目标为编辑器下单次 `parse` P95 在常见硬件上 &lt;100ms（随规格增大可调）。

## 10. 双产品线定位（M7 修订）

| 产品 | 职责 | 不在此仓库 |
|------|------|------------|
| **VS Code 扩展** | 文本编辑 + LSP + Hybrid **装饰** POC | FSF Webview、模块图 Webview、informal 表单 |
| **Electron + Vue 编辑器**（独立项目） | 块 UI、模块关系图、informal 富文本 | — |
| **本仓库** | Parser、LSP、`editor-api`、集成参考 | 生产块 UI、Electron E2E |

Vue 编辑器通过 npm 包消费：`@agile-sofl/parser`（检查）、`@agile-sofl/editor-api`（块模型/图/patch）、可选 spawn LSP（补全/hover/semantic）。
