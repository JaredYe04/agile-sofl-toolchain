# 09 — 测试与贡献

本文档说明如何运行测试、理解 fixture 布局、添加用例，以及通过 `inspect` / REPL 手动验证。

## 1. 测试基础设施

Vitest + TypeScript，无需先 build 即可跑测试。

```bash
npm test              # 全部测试
npm run test:watch    # 监视模式
npm run coverage      # 覆盖率（~85% 行覆盖）
npm run lint          # ESLint
npm run build && npm run cli -- inspect tests/fixtures/integration/banking.asfl
```

| 指标 | 数值 |
|------|------|
| 测试文件 | 19 |
| 测试用例 | 106+（82+ 起步，持续增长） |
| 行覆盖率 | ~85% |

低覆盖优先改进：`walk.ts`、`checker.ts`、`cstToAst.ts` 部分分支。

## 2. 目录与 fixtures

```
tests/
├── helpers/index.ts       # loadFixture, expectParseOk, expectDiagnostic
├── fixtures/
│   ├── grammar/module/minimal.asfl
│   └── integration/banking.asfl
├── lexer/                 # lexer.test.ts, lexer-boundary.test.ts
├── parser/                # modules, expressions, types, fsf-process
├── scope/                 # semantic, resolver-extra
├── typecheck/             # typecheck, types
├── fsf/                   # classifier
├── print/                 # roundtrip
├── ast/                   # guards
├── cli/                   # report, repl
├── integration/           # 多模块 + fixture 驱动
├── cli.test.ts            # spawn CLI 子进程
├── index.test.ts
└── smoke.test.ts
```

| fixture 路径 | 用途 |
|--------------|------|
| `grammar/` | 单一语法结构 minimal 样例 |
| `integration/` | 多特性组合的真实规格 |

```typescript
import { loadFixture } from '../helpers/index'

```

## 4. 添加测试

### 4.1 解析器测试

```typescript
describe('Parser - my feature', () => {
  it('parses valid construct', () => {
    const ast = expectParseOk('module SYSTEM_T; end_module', parse)
    expect(ast.type).toBe('program')
  })
  it('reports ASFL_PARSE_001', () => {
    const { diagnostics } = parse('module')
    expect(diagnostics.some(d => d.code === 'ASFL_PARSE_001')).toBe(true)
  })
})
```

### 4.2 语义测试

```typescript
import { check } from '../../src/index.js'

it('detects duplicate module', () => {
  const { diagnostics } = check(
    `module SYSTEM_A; end_module; module SYSTEM_A; end_module`
  )
  expect(expectDiagnostic(diagnostics, 'ASFL_SCOPE_002')).toBe(true)
})
```

### 4.3 Fixture 集成

1. 新增 `tests/fixtures/.../*.asfl`
2. 在 `integration/` 或对应模块测试中 `loadFixture` + `check()`

### 4.4 CLI / inspect

```typescript
const report = inspect(source, { tree: true, tokens: true })
expect(report.text).toContain('Agile-SOFL Inspect')
expect(report.exitCode).toBe(0)
```

`cli.test.ts` 通过 `spawnSync` 验证 `dist/cli.js` 退出码。

### 4.5 约定

- 文件：`*.test.ts`，目录对应 `src/`
- 测行为与诊断码，非实现细节
- 新语法：正例 + 关键负例

## 5. 覆盖率

```bash
npm run coverage
```

报告输出至 `coverage/`，可打开 `coverage/index.html`。目标：维持 ≥85% 行覆盖。

## 6. 手动验证

```bash
npm run build
node dist/cli.js inspect --tree --tokens draft.asfl
npx asfl repl
```

REPL 适合逐段粘贴、`:tree`/`:tokens` 切换、`:format` 查看规范化。

管道：

```powershell
"module SYSTEM_T; var x: nat; end_module" | node dist/cli.js inspect
```

手动通过后，将 `.asfl` 存入 fixtures 并补自动化测试。

## 7. 贡献流程

1. 从 `main` 建分支
2. 实现 + 测试（每个 fix/feature 至少一个用例）
3. 本地门禁：`npm run lint && npm test && npm run build`
4. 变更 API/诊断码/CLI 时同步 `docs/`
5. 提交信息简洁说明动机

## 8. 常见问题

**找不到 fixture？** 路径相对 `tests/fixtures/`，注意大小写。

**CLI 测试失败？** 确保 `npm run build` 生成 `dist/cli.js`。

**只跑单个文件？**

```bash
npx vitest run tests/parser/modules.test.ts
npx vitest run -t "duplicate module"
```
