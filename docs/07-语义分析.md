# 07 — 语义分析

本文档说明 `@agile-sofl/parser` 在 AST 之上的三项语义能力：**模块树与符号表**、**静态类型检查**、**FSF 形式/半形式分类**，以及 **信息隐藏**（`ext rd` / `ext wr`）。

## 1. 语义分析在流水线中的位置

```
parse → AST
         │
         ├─ resolveScope(program)   → ModuleScope 树 + ASFL_SCOPE_*
         ├─ typeCheck(program)      → ASFL_TYPE_*
         └─ classifyFsf(program)    → ASFL_FSF_*
```

`check(source)` 与 `inspect(source)` 在解析成功后依次调用上述三阶段，合并所有诊断。

## 2. 模块树（Module Tree）

### 2.1 模块声明与父子关系

```asfl
module SYSTEM_R;
type Base = nat;
end_module;

module Child / R;
type T = Base;
var x: T;
end_module
```

- `isSystem: true` 标记系统根模块（名称以 `SYSTEM_` 开头）。
- `parent?: QualifiedNameNode` 指向父模块名。
- `resolveScope` 为每个模块构建 `ModuleScope`，再通过 `linkParentChild` 建立 `parent` / `children` 链。

### 2.2 ModuleScope 与符号表

```typescript
interface ModuleScope {
  module: ModuleNode
  parent?: ModuleScope
  symbols: Map<string, SymbolEntry>  // kind: const|type|var|process|function
  children: ModuleScope[]
}
```

`buildModuleScope` 依次注册模块内 const、type、var、process、function。根模块默认为 `isSystem` 模块。

### 2.3 符号查找

`lookupSymbol(scope, name, moduleQualifier?)`：

- **无限定符**：沿 `parent` 链向上查找（子模块可引用父模块符号）。
- **有限定符**（如 `R.Base`）：在模块树中按名称匹配子模块或父链。

### 2.4 作用域诊断

| 场景 | 诊断码 |
|------|--------|
| 两个模块同名 | `ASFL_SCOPE_002` |
| 子模块写入父模块声明的变量 | `ASFL_SCOPE_003` |
| 引用未定义符号 | `ASFL_SCOPE_001` | `checkReferences` 遍历 AST 引用位；量词/comprehension/let 绑定、informal 文本、声明名排除 |

`checkVarWriteAccess(program, varName, writerModule)` 检查变量写入权限：变量只能在**声明它的模块**内被 `wr`。

## 3. 类型检查策略

实现位于 `src/typecheck/checker.ts`，内部类型见 `src/typecheck/types.ts`。

### 3.1 检查范围

对每个 `ModuleNode` 执行 `checkModuleTypes`（传入 `resolveScope` 的模块树）：

1. **类型别名**：`type` 段中每个 `typeExpr` 可解析；未解析的命名类型 → `ASFL_TYPE_002`
2. **变量声明**：`var` 的 `typeExpr` 必须 `isResolvedType`；否则 → `ASFL_TYPE_002`
3. **不变式 / FSF**：`checkPredicate` 检查关系式与调用；非 `bool` 表达式 → `ASFL_TYPE_001`
4. **进程参数组**：`inputs` / `outputs` 经 `validateTypeExpr`；未知参数类型 → `ASFL_TYPE_002`
5. **ext wr**：带注解的 `wr` 与变量推断类型比较 → `ASFL_TYPE_001`
6. **函数**：参数类型、`returnType` 可解析；有 body 时 `inferExprType` 与返回类型 `typesCompatible`；`== undefined` 跳过 body 检查
7. **常量 / 调用**：`checkExprCalls` 检测内建拼写（编辑距离 ≤1）→ `ASFL_TYPE_003`

### 3.2 类型环境与推断

`buildTypeEnv(module, scopes)` 沿 **父模块链** 合并祖先模块的 `type` 声明，再注册当前模块的 vars、consts、进程/函数参数。子模块（`Child / Parent`）可直接使用父模块类型名（如 `CustomerId`、`PriceKeys`）。

`inferExprType` 要点：

- 非负整数字面量：`0` → `nat0`，正整数 → `nat`；实数 → `real`；负整数 → `int`
- `if_expr` / `case_expr`：`unifyExprTypes` 合并分支；数值分支按 `nat0`⊂`nat`⊂`int`⊂`real` 加宽
- `let_expr`：局部绑定扩展 env 后推断 body
- `relational_expr`、合法内建 → `bool`
- 无法推断 → `unknown`（与 `unknown` 比较时 `typesCompatible` 视为通过，减少误报）

### 3.3 兼容性与诊断码

| 码 | 含义 |
|----|------|
| `ASFL_TYPE_001` | 类型不匹配（含函数返回、FSF 非 bool、`ext wr`） |
| `ASFL_TYPE_002` | 未知类型 / 未解析类型别名 |
| `ASFL_TYPE_003` | 非法或疑似拼错的内建函数名 |

`typesCompatible` 支持数值子类型、集合/序列/映射/积/composed/union/enum 结构规则。`typesCompatibleStrict` 用于函数返回、FSF 谓词、`ext wr` 等不允许 `unknown` 宽化的场景（已从 `@agile-sofl/parser` 导出）。`typeCheck(program, scopeResult?)` 可复用已有 `ScopeResult`，避免重复构建模块树。

## 4. FSF 分类器

实现位于 `src/fsf/classifier.ts`。

### 4.1 形式性判定

`atomIsFormal(atom)` 规则：

- `informal_text` → **非形式**
- `not_predicate` / `paren_predicate` → 递归检查
- `quantified`、关系式、表达式 → **形式**

对整个 `FsfSpecNode`（scenario 的 test/def 及 `others`）遍历，任一原子非形式则 FSF 为非形式。

### 4.2 进程层级与诊断

| 条件 | 诊断码 | 级别 |
|------|--------|------|
| 无 `decomposition`（底层）且 FSF 非形式 | `ASFL_FSF_001` | warning |
| 有 `decomposition` 且 FSF 全形式、无 comment | `ASFL_FSF_002` | info |
| 有 scenario 但无 `others &&` | `ASFL_FSF_003` | info |

**底层判定**：`process.body.decomposition` 不存在即为底层。

半形式典型结构（Banking fixture）：

```asfl
process A (...) ...
ext
rd a: int
wr b: nat
FSF :
  x > y && ... ||
  others && ...
decom: Banking_Decom
comment: informal note
end_process
```

FSF 谓词可全为形式表达式，`comment` 承载非形式说明。

### 4.3 函数 FSF

函数附 `fsf` 且非形式时报告 `ASFL_FSF_001`（warning，消息注明 semi-formal 可接受）。

## 5. 信息隐藏：ext rd / ext wr

Agile-SOFL 通过进程体 `ext` 段声明对外部模块变量的访问，实现信息隐藏。

```asfl
process P (...)
ext
rd balance: nat
wr log: seq of string
FSF : ...
end_process
```

AST 中每项为 `ExtVarNode`：`access: 'rd' | 'wr'`，可选 `typeExpr`。

| 访问 | 含义 |
|------|------|
| `rd` | 只读引用外部变量 |
| `wr` | 可写外部变量；受声明模块约束 |

类型检查对 `wr` 项验证类型兼容（`ASFL_TYPE_001`）。`checkVarWriteAccess` 禁止子模块 `wr` 父模块变量（`ASFL_SCOPE_003`）。

`VariableNode.kind` 区分 `normal` / `ext` / `ext_hash`，供后续访问控制扩展。

## 6. 集成基准

`tests/fixtures/integration/banking.asfl` 覆盖系统模块、const/type/var/inv、带 ext 与 FSF（含 others/decom/comment）的进程。对其执行 `check()` 应无 error 级诊断。

## 7. 扩展与局限

当前语义分析定位为**实用静态检查**，非完整形式化验证：

- 不做路径敏感或全程序数据流分析
- 复杂表达式推断保守返回 `unknown`
- 跨模块引用以模块树 + 名称匹配为主

LSP 集成复用 `ModuleScope`、`Diagnostic` 与 `Span`（见 [10-编辑器路线图.md](./10-编辑器路线图.md)）。`check()` 诊断（含 `ASFL_SCOPE_001`）经 language-server 缓存后发布为 LSP `publishDiagnostics`。
